# Rename Candidates Register

Purpose: track deferred service/class/file renames in one place so execution is fast when timing is right.

Linked governance docs:
- `docs/MASTER_PLAN.md`
- `docs/MASTER_PLAN_TRACKER.md`
- `docs/EXECUTION_BOARD.md`
- `docs/STATUS_WEEKLY.md`

Validation:
- `python3 scripts/validate_rename_candidates.py`
- `python3 scripts/suggest_rename_candidates_inventory.py` (advisory inventory for untracked `*Service`/`*Orchestrator` names)
- `python3 scripts/suggest_rename_candidates_inventory.py --fail-on-untracked` (required in CI; fails if inventory has untracked candidates)
- `python3 scripts/suggest_rename_candidates_inventory.py --apply --phase-close-target P10` (autofill untracked candidates into this register)
- `scripts/run_phase_rename_closeout.sh P#` (phase-close one-command autofill + validation)

How to use:
1. Add one row per rename candidate.
2. Set `category` (`service`, `orchestrator`, `class`, `method`, `feature`, `file`) and `phase_close_target` (`P#`) so phase-close gating is enforceable.
3. Keep `trigger` concrete (what condition means "do it now").
4. Link to the exact files/usages for quick execution.
5. Mark `status` as `planned`, `ready`, or `done`.

Policy:
- **Architecture-anchored only:** each entry must include `ideal_architecture_ref`.
- **Phase-close execution:** when a phase in `docs/EXECUTION_BOARD.csv` becomes `Done`, all rename entries targeting that phase (`phase_close_target`) must also be `done`.

Phase-close runbook (required):
1. Run closeout: `scripts/run_phase_rename_closeout.sh P#` (or run inventory manually if needed).
2. Set `target_milestone` + move eligible rows to `ready`.
3. Execute renames and mark rows `done` before phase status is set to `Done`.
4. Run guard: `python3 scripts/validate_rename_candidates.py`.
