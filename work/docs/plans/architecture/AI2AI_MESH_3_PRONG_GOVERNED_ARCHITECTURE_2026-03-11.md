# AI2AI + Mesh 3-Prong Governed Architecture

**Date:** March 11, 2026  
**Status:** Proposed architecture note  
**Purpose:** Reorganize AI2AI, meshing, and communications so they are properly split across the three prongs, governable inside AVRAI OS, and learnable by the reality model without collapsing prong boundaries.

---

## Executive Position

Yes, AVRAI should move toward:

- a **Mesh Governance Kernel**
- an **AI2AI Governance Kernel**

But they should not be implemented as two isolated new services that bypass the current kernel doctrine.

They should be implemented as **kernel families across the three prongs**:

1. **Apps Prong**
   - simple controls and truthful status surfaces
2. **Runtime OS Prong**
   - authoritative execution, policy, transport, privacy, and governance
3. **Reality Model Prong**
   - learnable priors, outcome scoring, and adaptive recommendations for mesh/AI2AI behavior

That means:

- runtime governs
- reality model learns
- apps project and control

This is the cleanest way to make AI2AI and meshing:

- governable in the OS
- learnable from on-device outcomes
- understandable in admin
- simple in the user product

The main architectural rule is:

**Mesh and AI2AI should become first-class governed domains, but they should still express themselves through AVRAI's existing kernel-native observation surfaces: `who`, `what`, `when`, `where`, `why`, and `how`.**

They should not become ad hoc seventh and eighth top-level kernel surfaces.

---

## Why Reorganization Is Needed

The audit showed a consistent structural problem:

- mesh and AI2AI are present in many places
- but they are not organized as one cross-prong architecture

Today:

- runtime owns too much of the behavior in a monolithic orchestration cluster
- reality-model learning from mesh and AI2AI exists only in partial, indirect, or local subsystems
- apps expose settings, feeds, and admin views without a fully canonical transport truth underneath

This creates three long-run problems:

1. **Governance drift**
   - transport, AI2AI learning, trust, and privacy are enforced in different places
2. **Learning drift**
   - the reality model sees some mesh/AI2AI effects, but not as a first-class learnable system
3. **Product truth drift**
   - users and admins see simplified outputs, but those outputs are not yet powered by a single canonical lifecycle

Reticulum helps with the runtime/control-plane side.
Signal helps with the security/session side.
Sideband helps with operator and device-router product patterns.
Haven helps with gateway and deployment layering.
ATAK helps only with admin visualization patterns.

The right AVRAI move is to absorb the useful structure from all of them into one governed three-prong design.

## Canonical Domain Split

AVRAI should separate these domains more cleanly.

## 1. Mesh

Mesh is the **transport substrate**.

It owns:

- bearer discovery
- destination reachability
- path selection
- path freshness
- custody transfer
- queueing/store-carry-forward
- relay/gateway segmentation
- transport health and recovery

Mesh does **not** own:

- message meaning
- chat semantics
- AI learning value
- recommendation content

Reticulum is the strongest architectural reference for this layer.

## 2. AI2AI

AI2AI is the **agent communication and learning domain** built on top of mesh.

It owns:

- peer identity binding
- session bootstrap
- prekey exchange and forwarding
- trust state
- payload classes
- delivery/read/learning semantics
- learning-eligibility policy
- peer-derived model update intake

Signal is the strongest architectural reference for secure session behavior in this layer.

## 3. Communications

Communications is the **product and admin projection layer** of AI2AI + mesh.

It owns:

- user-visible chat state
- user-visible "AI learned" state
- network wisdom projections
- admin topology and route visibility
- operational health summaries

Communications should not own routing or model logic.
It should project the truth generated by mesh + AI2AI.

## Current Placement Problems Across the Three Prongs

## 1. Apps Prong

Apps already expose AI2AI-related controls and outputs, but they are projections over fragmented internals.

Examples:

- privacy toggles for AI2AI learning
- network wisdom feed
- admin AI2AI dashboard and communications snapshots

Problem:

- app surfaces do not yet consume a single canonical transport + AI2AI lifecycle

What that means:

- UX simplicity exists
- architectural truth underneath is not yet unified

## 2. Runtime OS Prong

Runtime currently contains the bulk of mesh and AI2AI behavior.

This is correct in principle, but too much is concentrated in orchestration-heavy classes and legacy transport paths.

Examples:

- `VibeConnectionOrchestrator` mixes discovery, BLE startup, trust, activation receipts, loop protection, event mode, cloud sync, and learning throttles
- deprecated `AI2AIProtocol` still sits in mesh paths
- ACK, queueing, anonymous comms, trust, and telemetry are split across separate services

Problem:

- runtime owns execution, but not through a clean domain control plane

## 3. Reality Model Prong

The model side already learns from some mesh/AI2AI effects, but only partially.

Examples:

- locality inference blends mesh neighbor deltas into state
- embedding delta collection exists
- federated-learning codec exists
- AI2AI chat analysis and recommendation logic exist

Problem:

- there is no canonical "mesh world state" or "AI2AI outcome state" learned as first-class model truth

So AVRAI currently has:

- transport behavior in runtime
- learning fragments in runtime/model-adjacent services
- user/admin projection in apps

What it lacks is a clean, governed, learnable architecture joining all three.

## Proposed 3-Prong Organization

## Apps Prong: Projection + Control

Apps should own only:

- user consent and participation controls
- privacy mode selection
- simple delivery/read/learning status
- admin topology, queue, route, custody, and kernel-state visualization

Apps should not own:

- route scoring
- trust scoring
- message lifecycle policy
- learning-eligibility policy
- transport retry logic

Design rule:

- apps are the shell
- runtime is the authority
- engine is the learner

## Runtime OS Prong: Authority + Execution

Runtime should become the canonical home of:

- mesh control plane
- AI2AI session and payload lifecycle
- privacy mode enforcement
- trust/custody gating
- route receipt generation
- activation receipts
- admin-operable kernel state and lineage

This is where Mesh Governance Kernel and AI2AI Governance Kernel belong as executable policy and runtime authority.

## Reality Model Prong: Learning + Adaptive Guidance

Reality Model should learn:

- which path choices succeed
- which custody patterns are reliable
- which peer/interaction types produce useful learning
- which AI2AI exchanges improve downstream outcomes
- which network conditions justify different runtime suggestions

Reality Model should not:

- directly route packets
- directly mutate runtime state without governance events
- become an undocumented transport backchannel

It should emit:

- scores
- priors
- confidence
- policy recommendations
- anomaly convictions

Runtime decides whether to act on them.

## Shared Prong: Contracts

Shared contracts should carry the canonical cross-prong truth for:

- route receipts
- delivery lifecycle
- AI2AI learning outcome tuples
- trust and custody summaries
- kernel governance receipts

Without this shared layer, the three prongs will drift again.

## The Kernel Family Model

The cleanest design is not two isolated kernels.
It is **two kernel families**, each with runtime authority and model-learning counterparts.

## 1. Mesh Governance Kernel Family

### Runtime Kernel

Suggested kernel:

- `urk_mesh_runtime_governance`

Primary home:

- Runtime OS prong

Responsibilities:

- destination announcements
- path table management
- bearer scoring
- custody transfer rules
- queue expiry and retry policy
- route-health and path-churn tracking
- relay and gateway trust segmentation
- private mesh policy enforcement

Reticulum-derived ideas to adopt here:

- destination-based reachability
- reusable path tables
- store-carry-forward
- authenticated segmentation
- explicit delivery/custody proofs

### Model Twin

Suggested kernel:

- `urk_mesh_learning_intake`

Primary home:

- Reality Model prong

Responsibilities:

- learn path reliability priors
- learn route failure signatures
- learn delivery probability by bearer/mode/context
- learn locality propagation patterns
- recommend path policy adjustments inside governed bounds

Learned input examples:

- direct delivery rate
- relay custody success rate
- queue age before delivery
- route expiry frequency
- path churn
- latency by privacy mode
- battery cost by bearer
- prekey bootstrap success by route type

### App Projection

Apps should surface:

- `sending`
- `delivered`
- `read`
- `AI learning updated`

Admin should additionally surface:

- direct vs relayed vs delayed
- queue depth
- path churn
- custody handoffs
- kernel state and health

### Why This Family Helps

- gives runtime a single transport authority
- gives the model a real transport learning signal
- keeps route complexity out of normal UX
- makes private mesh operations governable and observable

---

## 2. AI2AI Governance Kernel Family

### Runtime Kernel

Suggested kernel:

- `urk_ai2ai_runtime_governance`

Primary home:

- Runtime OS prong, with governance-core scope

Responsibilities:

- peer/session eligibility
- payload class policy
- prekey forwarding policy
- trust-level gating
- learning-eligibility gating
- delivery/read/learning lifecycle semantics
- cross-peer privacy guarantees
- fail-closed behavior when crypto/bootstrap state is unsafe

Signal-derived ideas to preserve here:

- session bootstrap with strong prekeys
- forward secrecy
- rekeying
- skipped-message recovery
- explicit separation of transport failure vs session failure

### Model Twin

Suggested kernel:

- `urk_ai2ai_learning_governance`

Primary home:

- Reality Model prong

Responsibilities:

- learn which AI2AI interactions are actually useful
- distinguish noisy peer influence from meaningful peer influence
- score learning value before applying personality or locality updates
- detect harmful or low-value AI2AI loops
- recommend tighter or looser intake based on observed outcomes

Learned input examples:

- learning applied success rate
- downstream recommendation improvement after AI2AI updates
- trust score drift
- repeated low-value peer exchanges
- quality of cross-personality deltas
- locality uplift after imported mesh deltas

### App Projection

Apps should surface:

- AI2AI participation control
- privacy mode summary
- AI learning effectiveness summary
- human-readable network wisdom

Admin should surface:

- peer trust health
- AI2AI learning yield
- rejected vs accepted learning intake
- prekey/bootstrap failure classes

### Why This Family Helps

- AI2AI becomes a governed cognitive domain instead of a bag of services
- learning from peers becomes measurable and policy-bounded
- runtime and model roles stay separate
- the system can improve AI2AI quality without exposing raw complexity

---

## 3. Communications Projection Layer

Communications should not be a third governance kernel.

It should be a projection layer built from the two kernel families above.

User-visible communications should be derived from:

- Mesh Governance Kernel truth
- AI2AI Governance Kernel truth

This layer should own:

- chat projections
- feed projections
- notification semantics
- admin replay and topology views

It should not own:

- transport policy
- trust policy
- learning acceptance policy

This keeps the product clean and honest.

## How This Fits the Existing Kernel Doctrine

Mesh and AI2AI should not bypass the six canonical kernel surfaces.

Each kernel family must expose:

## `who`

- which node, peer, relay, or agent was involved

## `what`

- what payload class, route class, or transport event occurred

## `when`

- message age, queue age, TTL, delivery time, read time, learning-apply time

## `where`

- locality scope, mesh segment, private mesh boundary, gateway reachability

## `why`

- why a route was chosen
- why a learning update was accepted or rejected
- why a peer was trusted, quarantined, or downgraded

## `how`

- bearer used
- execution path
- encryption/session path
- retry/custody/recovery path

This matters because AVRAI doctrine already says governance must be kernel-native and bidirectional.

So:

- mesh/AI2AI are not outside the kernel system
- they are domain-specific governed kernel families inside it

## Bidirectional Flow for Mesh + AI2AI

The existing bidirectional governance note maps well to this design.

## Outward Flow

Canonical shape:

`app event -> runtime kernels -> mesh/ai2ai lifecycle -> shared contract -> reality-model learning -> upward governance summaries -> admin`

Examples:

- message delayed
- custody transferred
- read receipt observed
- learning applied
- path degraded
- prekey bootstrap failed

These become:

- runtime receipts
- learning tuples
- higher-order governance summaries

## Inward Flow

Canonical shape:

`admin or higher-order governance -> reality-model recommendation or signed directive -> runtime kernel action -> app-visible effect`

Examples:

- quarantine a relay class
- lower confidence in a peer cohort
- tighten AI2AI learning intake threshold
- prefer private mesh over cloud relay in a locality
- disable a broken route policy

This must happen through explicit governance events, never through hidden mutation.

## What to Move Where

## 1. Pull mesh control-plane ownership out of deprecated protocol paths

Do not let future pathing, custody, or queue truth grow inside the deprecated packet path.

Move to:

- Mesh Governance Kernel family in runtime

## 2. Break up the orchestrator monolith

`VibeConnectionOrchestrator` should stop being the implicit home of:

- discovery
- route policy
- trust
- event mode
- cloud sync
- learning throttles
- activation receipts

It should become a coordinator over explicit kernel-governed subdomains.

## 3. Turn locality mesh support into a formal `where` adapter

Current locality mesh updates are already flowing through runtime into locality inference.

That should become explicit:

- mesh is a transport source
- `where` / locality is the learned spatial consumer

This is a strong existing pattern worth preserving.

## 4. Extend route receipts instead of inventing another truth object

`TransportRouteReceipt` should become the canonical receipt for:

- queued
- custody accepted
- released
- delivered
- read
- learning applied
- expired
- quarantined

That lets runtime, model, and admin share one lifecycle vocabulary.

## 5. Move AI2AI learning from "chat analyzer feature" to "governed learning intake"

Current AI2AI learning helpers are useful, but they should not be the canonical architecture center.

Canonical center should become:

- runtime emits governed AI2AI outcome tuples
- reality model scores and learns from them
- apps consume the projections

## 6. Treat `NetworkWisdomFeed` as a projection surface, not a source of truth

Its role should be:

- user-friendly feed over canonical AI2AI + mesh receipts and outcomes

Not:

- an implicit learning or transport authority

## Suggested New Kernel Registry Additions

The current registry already provides good anchors:

- `urk_stage_c_private_mesh_policy_conformance`
- `urk_learning_update_governance`
- `urk_user_runtime_learning_intake`
- `urk_kernel_activation_engine`
- `urk_kernel_control_plane`

The next step is to add explicit mesh and AI2AI kernel entries rather than leaving their behavior spread across services.

Recommended additions:

1. `urk_mesh_runtime_governance`
   - prong: `runtime_core`
   - privacy modes: `local_sovereign`, `private_mesh`, `federated_cloud`
   - locality scope: `device`, `mesh`, `cloud`
   - triggers: `mesh_path_update`, `custody_event`, `queue_expiry`, `runtime_health_breach`

2. `urk_mesh_learning_intake`
   - prong: `model_core`
   - privacy modes: `local_sovereign`, `private_mesh`, `federated_cloud`
   - locality scope: `device`, `mesh`, `cloud`
   - triggers: `route_outcome_observed`, `delivery_lifecycle_closed`, `path_policy_candidate`

3. `urk_ai2ai_runtime_governance`
   - prong: `governance_core`
   - privacy modes: `local_sovereign`, `private_mesh`, `federated_cloud`
   - locality scope: `device`, `mesh`, `cloud`
   - triggers: `ai2ai_message_candidate`, `prekey_forward_candidate`, `trust_change`, `policy_violation_detected`

4. `urk_ai2ai_learning_governance`
   - prong: `model_core`
   - privacy modes: `local_sovereign`, `private_mesh`
   - locality scope: `device`, `mesh`
   - triggers: `learning_update_candidate`, `learning_applied`, `peer_quality_drift`, `low_value_loop_detected`

These should integrate with:

- activation receipts
- kernel control plane
- lineage and admin observability
- self-learning governance

## Bottom Line

The clean architecture is:

- **Mesh as a governed transport domain**
- **AI2AI as a governed agent communication and learning domain**
- **Communications as a projection layer over both**

And the clean implementation pattern is:

- **runtime kernel authority**
- **reality-model learning twin**
- **app projection shell**

So yes, AVRAI should have a Mesh Governance Kernel and an AI2AI Governance Kernel.

But for the architecture to stay correct, each one must be implemented as a **three-prong family** rather than as a runtime-only singleton.
